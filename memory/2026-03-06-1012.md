## 什么时候该创建新 agent？

判断标准：

1. **任务可以完全独立完成**  
   输入清晰，输出明确，中间不需要频繁交互  
   ✅ 例子：批量处理 100 个 GitHub issues，每个 issue 一个 agent  
   ❌ 反例：你的编程任务（需要频繁协作）

2. **需要隔离的长期记忆**  
   不同角色的风格、偏好、历史记录应该分开  
   ✅ 例子：博客写作 agent（我）vs 编程 agent（coder）  
   ❌ 反例：同一个项目里的不同角色（应该共享项目记忆）

3. **并行处理**  
   多个独立任务同时进行  
   ✅ 例子：同时爬取 10 个网站，每个网站一个 agent

4. **安全隔离**  
   某些任务需要限制权限  
   ✅ 例子：只读数据分析 agent vs 可写操作 agent

## 你的编程任务应该怎么做？

**单 agent + 多工具/技能**

- 一个 agent 负责整个项目
- 用不同的工具/技能来完成不同阶段：
  - 需求分析 → 读文档、问用户
  - 写代码 → exec、edit、write
  - 测试 → exec 运行测试
  - 文档 → write README

这样：
- 上下文在一个会话里，不会丢
- agent 可以灵活切换角色（"现在我来测试一下"）
- 发现问题可以立即回到代码修改

## 多 agent 编排的真正用途

不是"角色分工"，而是：

1. **任务分发** → gh-issues skill 就是这样：主 agent 分配任务，每个 issue 一个子 agent
2. **专业化隔离** → 博客写作（我）vs 编程（coder），长期记忆和风格完全不同
3. **并行加速** → 同时处理多个独立任务

## OpenClaw 的实际约束

- 子 agent 通过 sessions_send 通信，不是共享内存
- 每次传递上下文都有 token 成本
- 如果需要频繁交互，单 agent 更高效

## 上下文爆炸的场景

**会爆炸的情况：**
1. **长期项目，持续积累**
   - 几周的开发，每天的对话都在同一个会话
   - 代码改了又改，历史版本、讨论、决策都堆在上下文里
   - 最终超过模型的 context window

2. **大型代码库**
   - 需要理解整个项目结构
   - 多个文件的代码都要加载到上下文
   - 很快就会超限

3. **复杂的多步骤任务**
   - 需求分析 → 设计 → 实现 → 测试 → 文档
   - 每一步的输出都成为下一步的输入
   - 上下文线性增长

## 但这不是"单 agent vs 多 agent"的问题

**关键洞察：**
- 多 agent 编排**并不能解决上下文爆炸**
- 因为：
  - 如果任务高耦合，你还是要把上下文传给每个 agent
  - 传递上下文 = 把信息塞进消息 = 还是占用 token
  - 甚至更糟：信息在传递中会衰减、需要重复解释

**真正的解决方案：**

### 1. **分阶段会话（Session Segmentation）**
不是拆 agent，而是拆会话：
- 需求分析阶段 → 会话 A，输出：需求文档
- 设计阶段 → 会话 B，输入：需求文档，输出：设计文档
- 实现阶段 → 会话 C，输入：设计文档，输出：代码
- 测试阶段 → 会话 D，输入：代码，输出：测试报告

每个阶段：
- 有明确的输入（文档/文件）
- 有明确的输出（文档/文件）
- 上下文重置，只加载必要的输入

### 2. **外部记忆（External Memory）**
不把所有东西都放在对话上下文里：
- 需求 → 写到 `requirements.md`
- 设计决策 → 写到 `design.md`
- 代码 → 写到文件系统
- 测试结果 → 写到 `test-report.md`

需要时再读取，而不是一直放在上下文里。

### 3. **增量式工作（Incremental Work）**
不要一次性完成整个项目：
- 先做核心功能（会话 1）
- 再做扩展功能（会话 2，只加载核心代码）
- 最后做优化（会话 3，只加载相关部分）

### 4. **工具辅助（Tool-Assisted Context）**
- 用 `read` 工具按需加载文件，而不是一次性全部加载
- 用 `exec` 运行测试，而不是在对话里模拟
- 用 `web_search` 查资料，而不是让模型回忆

## OpenClaw 的实际能力

- **sessions_spawn** 可以创建新会话（分阶段）
- **memory/** 文件系统可以作为外部记忆
- **skills** 可以封装复杂流程，减少对话复杂度

## 回到你的问题

**单 agent 多工具会上下文爆炸吗？**
- 如果你在**一个会话里完成整个长期项目** → 会
- 如果你**分阶段、用外部记忆、增量工作** → 不会

**多 agent 编排能解决吗？**
- 如果任务高耦合 → 不能，反而更糟
- 如果任务低耦合 → 可以，但本质是"分阶段会话"

## 我的建议

对于编程任务：
1. **不要拆成多个角色 agent**（路由、编码、测试、文档）
2. **拆成多个阶段会话**（需求 → 设计 → 实现 → 测试）
3. **用文件系统作为阶段间的接口**（不是消息传递）
4. **每个阶段的 agent 只加载必要的上下文**

这样：
- 避免上下文爆炸
- 保持任务连贯性
- 可以灵活回溯（重新开一个会话，加载之前的输出）
